Abstract:

Code quality is important for maintainability and sustainability of software systems, and is addressed by guidelines and peer reviews. There are well-defined methods and standards for code quality for example the Clean Code approach [1], and they are integrated with current software development process models such as SCRUM. However, developers have to be encouraged to invest effort due to the trade-off between software quality measures and projects' time pressure. Unfortunately, there are no proven hypotheses about the reasons for missing acceptance of such methods and standards by developers. To investigate hindering aspects for code quality acceptance in practice, empirical studies might help to determine the root problems and to establish solutions. However, there is a lack of empirical studies concerning this topic. Therefore, we performed an empirical study with practitioners to investigate problems and challenges while enforcing Clean Code. Semi-structured interviews were used to consult experienced practitioners from different companies and backgrounds. The transcripts were analyzed with open coding. The results showed that a lack of even one of the people-related aspects such as motivation, knowledge, experience and code quality awareness, hinders the enforcement of code quality. Surprisingly, tools might establish hindering factors due to high initial effort. As conclusions we propose implications for academic research, organization, practitioners and tool providers.

SECTION I.

Introduction

Maintainability can be achieved through readability, simplicity, testability and structuring [1], amongst others. Maintenance is an ongoing topic not just for practitioners but also in research as it plays an important part in software engineering. Many metrics, tools and methods have been proposed that support developers in writing maintainable code. Approaches like Extreme Programming [2] and SCRUM [3] include methods like Test Driven Development and continuous communication between developers to improve the process. Quality models like the ISO standard [4] that define software quality, have been introduced. Additionally, a lot of metrics have been proposed, e.g. lines of code and length of method, in order to assess code complexity. There is literature proposing principles and guidelines. The well-known book “Clean Code - A Handbook of Agile Software Craftsmanship” [1] provides a general overview on principles and methods on how to achieve good code quality. The word Clean Code is used to describe code that implements these principles and methods. We choose the book as a reference for code quality because it provides a comprehensive collection of best practices for code quality. This book is also well-known in the software development community. The aspects include the topics of naming, documentation, comments, formatting, functions, classes, error handling, boundaries, testing, code smells and refactoring.

Nevertheless, there are still challenges that are not addressed by research. Many projects suffer from low code quality and consequently technical debt [5], although research proposes and develops a lot of means for code quality assessment and enforcement. This may have several reasons. The code quality models are not very specific and do not include recommendations how to actually assess and improve quality. Furthermore, code quality metrics for assessment are only partially helpful and are usually project specific. The interpretation is often not clear and applicable values for thresholds are hard to identify. The lack of knowledge in development teams is another challenge. Furthermore, a missing awareness for code quality and its low priority is an issue. Such challenges can not solely be addressed by introducing more tools and methods.

In order to better understand the adoption of code quality in practice, we conducted an empirical study with practitioners. The goal of this paper is to identify crucial factors that hinder the enforcement of code quality and reasons for missing acceptance of code quality principles. We used semi-structured interviews to consult novices and experienced practitioners from different companies and backgrounds.

SECTION II.

Objectives and Research Questions

Because we wanted to identify problems and challenges, we used semi-structured interviews with open questions to encourage the interviewees to talk freely. Nevertheless, we had some hypotheses. We believe that a missing awareness for code quality is an important factor and also the overall acceptance of the needed steps are a problem. Furthermore, the prioritization of code quality is not high enough and there could be a general knowledge deficit.

The knowledge aspect is addressed by the first research question. We wanted to find out if there are knowledge gaps and if there is a need for action.

SECTION III.

Study Design

A. Sampling

The interviewees were found over personal connections. They came from seven different companies. Only three participants were actually working in the same team. The experience ranged from five months to 36 years. Furthermore, the interviewees have different degrees and the size of the company size, the interviewees work in, ranges from 40 to over 140.000. Participant C works at a big university which makes it hard to specify the number of employees. An overview of all the interviewees can be seen in Table I.

B. Interview Guide and Interview Process

The interview guide is divided into four parts. The questions are structured according to the research questions. The first part is about the interviewee's background to find out if there is a possible connection of experience or programming languages to the answers and to ensure transferability. The second part is about the individual interests and concerns of code quality. It is asked, what good and bad code meant to the participants and how they evaluate code. It also includes a question about confidence in evaluating code and if the term “Clean Code” is known by them and from where. This part is mostly connected to RQ1. The third and biggest part concerns the enforcement activities that are necessary to achieve good code quality. Here the actual workflow in the companies is inquired and the used guidelines, tools and other steps like testing and code review are further investigated. It is asked which methods for assuring code quality are used and whether the developers find them helpful. The last part is about the company's support of refactoring and further training. The interview finishes with the question if the participant misses or wishes for anything that is not already being used for supporting code quality evaluation.

Every participant was interviewed once for approximately one hour. Some of the voluntary interviews took place in the companies, over the Internet or at the developer's home. The interviews were recorded by smartphone and notebook. Additional notes were taken after the interview was finished, because many of the interviewees made some interesting points after the recording was stopped. The interview was tested once to ensure that the questions are understandable.

Open coding was used for qualitative data analysis of the transcripts. SECTION IV.

Study Results

In the following, the findings are presented. It has to be noted that the interviews were held in German and the quotations are translated into English. A quotation is followed by the respective participant and the used code.

  • RQ1:

    Which guidelines, principles, conventions and other aspects concerning code quality are known by developers?

Comprehensibility and readability were often named first and most often by the participants when asked for a definition of clean code. In connection to them structuring, naming and documentation were mentioned often. Structuring was used as an umbrella term for modularization. This resulted in a mind map that connects the different named aspects (see Figure 2) and include their corresponding participants. The branches of the mind map can be read like a causal chain. For example, when the length of a function is limited, the units get smaller. This helps with the structuring and consequently leads to more comprehensibility. These casual chains were derived from participants' statements. The mind map is not complete, because the participants were not explicitly asked for these connections. The visual aspect of code comprehension was emphasized numerous times: “Good code is basically code, that immediately appeals to me visually” (participant A, “appealing appearance”). It was also said that debugging is used as a means to understand code: “When I read the code and I don't [understand it] immediately, then I have to somehow run it, I have to trigger the part and debug through it, then I often understand what is happening there” (participant L, “debugging to help understand code”). That is probably why many participants emphasized easy “entry points” (participants A, C, G, H) and “speaking names” (participants A, B, C, D, F, G, I, L). It seemed easier for the participants to answer, what bad code is to them, i.e. what good code should not look like: “[…] in very very complicated code, with many lines, with many things, I don't know my way around, it is less maintainable, it is more fault-prone, it is not easily changeable and so on” (participant I, “complexity”, “maintainability”, “extensibility”). When asked about bad and good code, code smells were named frequently without defining them as such. The following code smells were named: duplication, wrong abstraction, wrong naming, missing tests, side effects, high number of parameters, noise (e.g. unnecessary comments) and cycles. Duplication was mentioned as the most crucial code quality problem.

Documentation was mentioned repeatedly and by nearly all participants (11 out of 12) and was seen as an important factor for good code quality, too. The participants talked about comments, but also API and documentation of standard solutions. “The basis is the public API of a class. That is kind of important” (participant A, “API”).

Principles [1] like DRY (“Don‘t repeat yourself”, 1 out of 12), KISS (“keep it simple, stupid”, 1 out of 12) or even SOLID (2 out of 12) were rarely used to describe good code quality. They were referred to indirectly by saying code should not be unnecessarily complex (participants A, C, D, K, L) or there should be no duplication (participants C, D, E, F, G, K, L). In the case of SOLID, Single Responsibility which is one of the underlying principles, was mentioned the most with 4 out of 12 participants. It is not clear, if principles are not helpful because they summarize things developers already know or if maybe the definitions are not as clear as they should be. Concepts like architecture and design patterns were named, too. It is not clear where the boundaries between code quality and architecture are: “that's were I'm not sure if this is still rather code quality or is it already architecture?” (participant L, “architecture”).

The mentioned aspects were often not introduced at university but by colleagues, the company or learned while educating oneself as stated by the participants. The preferred source of information were literature, blogs and forums, but also conferences and workshops.

  • RQ2:

    How is Clean Code implemented and ensured in practice?

Here, the used steps, means and tools used to achieve good code quality are illustrated. The aspects from RQ1 can be seen as enforcement goals. The goal is to automate as much as possible to speed up the development process: “You should automate as much as you can” (participant E, “automating”).

Conventions and Guidelines

Documented conventions and guidelines in wikis and similar are not always seen as helpful, because they are too long (four participants), not everyone knows them (two participants), they are not being followed (two participants) or they are not easily controllable (two participants). Only 4 teams have documented conventions and guidelines in addition to static code analysis tools from which one even said, nobody is using them. The established conventions of the used programming language are commonly accepted by developers. One team used an explicit definition of done to ensure not only that the functionality but also the documentation and tests were written, too.

Processes

Table III shows the processes used by the par- ticipants in their company. The terms “integration tests” and “system tests” were used interchangeably by practitioners. For example, Selenium tests were seen as integration tests for one interviewee and for the next they were system tests. B, E and G are summarized in one line because they work in the same team and gave the same answers. It also has to be noted that in K's company code review is not mandatory anymore when pair programming is conducted.

Tests were not always an obligatory step and are skipped when the time pressure was too high. To ensure tests are done, test driven development (TDD) [8] is used in one team and two other developers use it occasionally. “[T]he result was good structured code” (participant D, “test driven development”). Behavior driven development (BDD) [9] and domain driven development (DDD) [10] were also mentioned, but nobody proceeds according to the principles of those methods. To make the process of controlling tests easier, Jenkins or a similar tool is used to automatically run the tests when a new commit was pushed or all integration/system tests are run over night. But there is still a lot of manual testing being done, especially before a release. In 8 out of 10 teams, manual testing is done on a regular basis. One of the other teams said, they have a lack of front end tests which indicates manual testing, too. Code coverage is used as an indicator but it often does not have any significant impact or is a metric for the management. When writing the test, it can be helpful to make sure every line of code is executed.

“Pair programming is used when problems occur” (par-ticipant B, “pair programming”), but not necessarily always. Usually, the teams switch the pair from time to time. For example, in one team they switched daily, in another one they switched based on the task. In case one person can not make progress, pair programming is used and a usually more experienced programmer supports them. It increases knowledge transfer and results in less errors in the code: “It's an ongoing process, you are learning all the time because you are doing pair programming” (participant I, “pair programming”). Pair programming also helps to decide refactorings without involving the whole team.

While developing, legacy code is marked for refactoring or small adjustments are done on the fly. “Refactorings simplified ensuring extensibility” (participant I, “refactoring ensures extensibility”). The developers knew the reasons for refactorings and found them important, but it is not always clear, how often big refactorings are being performed. One participant clearly said, they are not given time for refactorings and can only improve small parts of the old code while implementing a new feature or fixing a bug.

Code review is used like a quality gate in many teams (see Table III). The code has to be checked before it can be merged into the main branch. But this step is often not supported by any kind of checklist or defined standard: “I have the feeling that it is all done very subjectively. Everyone has their own guidelines according to which he is checking the code” (participant E, “different prioritization”). The reviewers have to decide for themselves if the code is good enough. Only one team uses a defined checklist for the reviewing process.

Code Quality Skills

To be able to execute the necessary steps, the developer needs knowledge and experience. There are often new team members that learn from experienced developers. There are four aspects that play a role when it comes to members of the team: experience, knowledge, code quality awareness and motivation. The connection of the four aspects can be seen in Figure 3. Experience is needed to be able to improve the code and experience often implies knowledge. The developer has to know the aspects that are important for code quality. The developers would not make the effort, if they do not have code quality awareness and know the benefits. Awareness helps with motivation but is not enough to ensure that all the necessary steps for code quality are done. Motivated developers are again more likely to practice more, which leads to more experience again.

Table III Interviewees’ USED processes

To build up knowledge and experience, further training is used. It is done on the initiative of the developers and there is not always reserved time in the working hours. Pair programming and code review can also play a big part in knowledge transfer and building up awareness. They are especially beneficial for new team members who have to get to know the code base. The company has to support the developers, too, to increase the developer's motivation to devote their time on further training themselves. A possible solution is to make quality steps like code review or refactoring obligatory: “It is important that a culture for Clean Code exists” (participant F, “awareness of Clean Code”).

“The best thing to achieve good code quality, is actually communication” (participant I, “communication”). All of the participants underline how important communication and discussion is for code quality: “If you are a little bit unsure, you can ask more experienced colleagues or even the architect […]” (participant F,“communication”). “One person rarely does everything by himself, there is much discussing and approaches are often talked over” (participant F, “discussion”, “finding consensus”). Two companies even have so-called guilds and communities of practices where interested practitioners from different teams in the company meet regularly to discuss and decide refactorings, tools and similar things: “That is of course the comfortable part, that it was discussed and defined somewhere and than you don't have to argue with your colleagues anymore” (participant F, “comfortable to have defined standard”). Moreover, approaches are discussed in the team to ensure the best possible solution. As emphasized by the participants, experienced developers make less mistakes and do not need explicit checklists which is another indication of how important experience is.

Tools

Every team has tools to support code quality control.

In 3 out of 10 teams a couple of the tools are obligatory from management. Otherwise new tools are rarely introduced by the company, “but rather one person just looked into it a little bit” (participant F, “finding new tool”). Decisions concerning tools and guidelines are made in the team. Only if the tool is not for free, the management is involved. Supporting tools in general are viewed as helpful if they are configured the right way: “[…] then all of them [are helpful]. First, they decrease the workload and second, they also lead to good readable code” (participant B, “configuration of new tools”). Jenkins or Git hooks are used to ensure that code analysis tools are executed regularly after each commit. The integrated tools that are included in IDEs like IntelliJ and Eclipse are popular, too. In Table IV the participants are linked to their named used tools. Since it was not especially asked about specific tools, it could be that some participants forgot a tool that plays part in helping the development process or that for some developers tools like ticket systems are not linked to the topic of code quality. An interesting part that can be noted is that teams in the same company do not necessarily work the same way or use the same tools. Sometimes even in the same team the knowledge or perception on how things are done is different.

  • RQ3:

    Which challenges occurred during the enforcement of code quality?

Code quality is not an easy topic and requires extra effort to accomplish: “[1]t is difficult. You can often easily say that it's not good code. But to say what is good and what is very good code, that depends very much on the situation” (participant C, “depends on context”). There is not always a right and wrong and “[…] other things are just personal taste” (participant B, “different prioritization”). Sometimes, aspects have to be weighed against each other. It is overall hard to balance cost efficiency when no direct benefit can be seen, it all depends on the context and aspects are subjective, too.

Table IV These are the mentioned and used tools by the participants.

The named challenges are ordered into three groups depending on the type of challenge: organizational, tool-related and people-related. Explicitly mentioned problems were taken into account, but some were also deducted from wishes and other statements made by the participants.

Organization

Code quality measurements are often started late and the technical debt [5] is already high. The most common challenge is the time constraint which was named by all of the participants. It leads to missing refactorings, tests and code review. Additionally, there are many requests from clients and not enough developers available to handle those requests. Not all developers were officially given time to look into new tools and do not get to further train themselves in the working hours. There were only two teams were the team members got regularly time to investigate new tools or other technologies and similar. Coordination of workshops are not easy either as “[… ] they are often at the same time as a release, when we have little time” (participant G, “further training is hard to coordinate with release dates”). Even choosing appropriate further training was stated as difficult by two participants. That is the reason why it was not seen as positive that further training has to be done on one's own initiative.

Refactorings, when they are done at all, are often not clearly organized. They are sometimes not finished or not every similar code part is adjusted accordingly. This leads to obsolete code and wrong solutions that can later be copied. It also occurs that the refactorings do not happen at all: “It was often decided that the effort to generalize it, that you could use it in other projects, was simply not profitable” (participant C, “cost benefit”, “deciding refactoring”).

Checklists or guidelines are not seen as helpful: “You would have to check for every class and every method if any of the points have to be considered, at some point you may know the list by heart but it is somehow too much administration effort” (participant E, “checklist has to be efficiently controllable”). 8 out of 12 participants even said they “evaluate [code] by feeling” and not with the help of a checklist.

Documentation is a difficult topic, because it has to be structured and it should not be too extensive, otherwise no one will find what they are looking for. There were only two participants that regularly write documentation. Standard solutions are not common either. Four participants even wished for a defined standard.

The requirements are not always clearly defined before the beginning of the implementation. This makes it hard to decide the design before starting the implementation and leads to broken architecture and expensive refactorings later in the implementation process. There is also often no communication between different development teams in the same company even though they could help each other with tools and frameworks. Only two participants have meetings across different teams concerning software engineering topics.

Processes to support code quality are not obligatory in every team and are even missing in others (see Table III): “[We] are missing instruments and processes to support that in any kind of form, may it be code review or trying to write unit tests” (participant A, “missing processes and instruments”, “missing code review”, “missing tests”). In 3 out of ten teams tests are not obligatory and code review is only done in seven of them.

Tools

Tools are essential in ensuring code quality an automating the enforcement as much as possible. Even though the benefits are high, they are not always used. The costs are rarely the reason for not introducing new tools. As one of the participant said: “Costs were never really an argument” (participant K, “costs”), but the tools have to be justified. Tools for free are never a problem, there only has to be someone willing to put in the initial effort for setting it up. But the initial effort seems to be the biggest problem. The different options have to be compared and analyzed to be able to decide which tool fits the purpose best and the effort to then make the new tool work, is a hindrance. Concerning code analysis tools, there are too many warnings that discourage the usage (6 out of 12 participants): “I think it is a big initial effort. It means that someone has to really look into it, has to read the workflow, has to read tutorials and has to train the employees. That is a big effort at the beginning but I think it would be profitable really fast” (participant G, “initial effort”). The numerous warnings lead to developers ignoring the tools all together. Tools that take a long time to run are too inefficient and unusable due to the time constraint.

Tools from different vendors are sometimes an issue when updating. Frameworks can become obsolete or the newest versions are not compatible anymore. Some “[… ] have to pay attention to licenses” (participant L, “licensing”) which can exclude some tools that are not meant for industrial use. There are not even options for tools for some programming languages: “But static code analysis for Python doesn't exist” (participant K, “missing tools”).

People-Related

A big problem is the missing awareness for Clean Code not only in the team, but also in other departments of the company: “Pair Programming is often seen critically by managers and other people outside of the team and it has to be defended again and again” (partic-ipant D, “pair programming”). That is one way, how the management or people outside of the development team can be a hindrance. The management does not always understand why something takes more time or do not see the necessity of refactorings: “[T]he managers interfere with the work of the IT although they don't have the expertise to assess how much time something takes or if a refactoring is necessary” (participant I, “management interferes with development”). The code quality awareness increases with time and experience for some developers - “The awareness formed over the last two years that you have to pay attention to extensibility and platform functionality” (participant L, “awareness of Clean Code”, “extensibility”), but others do not seem to learn - “the believe in the effect didn't set in” (participant D, “awareness of Clean Code”). Some do not seem to be interested in improving their knowledge as one participant said: “It would be nice if everyone was also a little bit interested in his own progress” (participant E, “awareness of Clean Code”). This results in missing tests and lowly prioritized refactorings.

Finding consensus and deciding the right approach are made difficult by different prioritization and because criteria have to be balanced all the time. For one person naming is really important but the next developer thinks that “[… ] speaking names are nice but not too long, please” (participant C, “balance of Clean Code criteria”, “naming”). The same goes for other conventions like function length: “I personally find it better when a function is little bit longer but the logic is reasonably cohesive” (participant C, “balance of Clean Code criteria”, “length of function”).

The need for experienced developers is, amongst other things, important for knowledge transfer. Inexperienced developers have to be corrected early in code reviews. There has to be an open culture as some of the developers are scared of criticism and do not wish for code review. Different expertise and general knowledge and experience were pointed out as important for code review. There were also many people-related aspects named concerning code quality which led to important characteristics of team members: “It is important to have a mixed team concerning the experience where everyone is open to new things” (participant F, “mixed team“, “open minded“) The knowledge also plays part in understanding code and one participant said that “when you don't understand something because you don't know how to do it yet, then the code that you don't understand is not necessary bad just because you don't understand it” (participant E, “comprehensibility depends on knowledge, “missing knowledge”). Three participants would like to have external consulting so that someone external can point out the mistakes and the developer himself is not seen as a know-it-all. All participants had some kind of problem with conceptual integrity [11] which is a quality of a system where all the concepts and their relationships with each other are applied in a consistent way throughout the system. The missing existence of standard solutions could be a reason for that. Another problem is when developers leave the company and there is no documentation.

Motivation plays a big part too, given that many aspects like further training, refactoring, documentation and even testing were named to depend on the initiative of the team members. Sometimes “[… ] nobody feels like taking care of it” (par-ticipant H, “missing motivation”) or the developer is “[… ] not always as disciplined as [he] would like” (participant L, “laziness”). There has to be at least one team member who is motivated to take the time for refactoring: “I think it just needs someone who is a little bit motivated, that just simply says I am going to take a day for that” (participant K, “missing motivation”). The company is not always encouraging its employees: “The problem is always that when you say, I have a bug here that I believe is from the product then I often get the task and you think, great, I didn't want that, I just wanted to report it” (participant H, “remarks lead to more work”, “missing motivation”). When there is no motivation, code quality will get neglected: “A lot is also negligence. Sometimes you want to just finish a project fast. Giving thoughts to good code, that takes time and if you have something, you don't feel like doing and you can make it work fast, you do it fast” (participant K, “negligence”).

SECTION V.

Discussion

The lack of knowledge seems not to be a key factor for missing quality measurements. The participants name many of the aspects of the Clean Code book [1]. Even principles and tools are known by them. Aspects that can be controlled more easily due to metrics or similar, are more present, while vague characteristics like sustainability and reliability are only mentioned occasionally. However, knowledge is not enough to achieve an enforcement of code quality.

There are also some people-related concerns. If there are no developers who are interested in Clean Code and drive it, Clean Code does not become a topic in a team. The key aspects here are: code quality awareness, knowledge, experience and motivation. The teams with few measures to ensure code quality have a lack of Clean Code awareness. The interviewed participants are already motivated to write Clean Code, however, the challenge is to motivate reluctant programmers or even improve motivation of others. There is not much research concerning the four people-related aspects experience, knowledge, awareness and motivation. Further ways to improve these four aspects and ways for higher priority of them could be further investigated.

Experience is not always an implication for confidence regarding the topic of Clean Code. We could notice that less experienced developers talked less about the topic, leading to shorter interviews. Of course it is very important for knowledge transfer to have experienced and knowledgeable developers. Team members can benefit from their experience and knowledge. If there are only beginners in a team, there is no one to check and to give hints for improvement. New developers also rely on experienced team members for further training. Furthermore, experience is important to judge about vague aspects. If a team makes tool decisions by itself, knowledge is required. But even experienced practitioners with knowledge are no guarantee for good code quality. Other factors like team members, time and management are still hindering it. Companies should pay more attention to the skills of the developers they employ and how to keep motivating them to stay and to improve code quality.

Time pressure leads to multiple consequences: Refactorings are not done, tests are skipped and developers generally work less thorough: “Selenium is not really controlled because there is not enough time and developers” (participant B, “missing developers”, “system tests”, “time constraint”). It also decreases the developers' motivation to pay attention to Clean Code because it would require more effort to show determination and never skipping a quality step. Moreover, missing tests make refactoring more risky or even impossible. Features need to be delivered as fast as possible. This means that the code needs to be built easily and the developer should get an immediate feedback about code quality. The trade-off between time and quality is hard to decide. It is not always clear if a refactoring is cost-efficient or if pair programming saves time in the long run. Furthermore, people tend to return to former bad habits under time pressure, and skip quality steps. Even evaluating, selecting and introducing new tools may not be considered as important enough by a company or by practitioners themselves. Code quality should get a higher priority in companies, so that activities are not skipped anymore. Practitioners should perform helpful activities like code review and refactorings.

If a team can decide about guidelines by itself, they are more understandable and developers are more likely to follow them because “[…] each developer knows exactly why he is doing it” (participant G, “knowing reason for Clean Code criteria”). However, according to the interviews, the helpfulness of guidelines is questionable. They are often seen as too long or not easily checkable. Furthermore, they are hard to establish and not tangible enough. If there is a well-configured code analysis tool, it is difficult to find additional aspects that are not covered by the tool and can still be effectively evaluated. That is the reason why many teams do not use them and the developers evaluate the code quality according to their own preferences. This can effect the quality of code reviews and it makes the process harder to define. Existing guidelines should be reviewed with these aspects in mind. Researchers should evaluate the effectiveness of guidelines together with and without static code analysis tools.

Pair programming is not always considered positively and it is stated as exhausting to perform it completely. The question is why it is so exhausting. Maybe it requires more attention from developers if someone else is looking over one's shoulder, and the need for a justification of decisions leads to a more thorough way of working in pairs. It may help to raise motivation to write cleaner code. Practitioners should evaluate in which cases pair programming helps them and should not mindlessly do it all the time or not at all either.

There are also indications that code quality should be a more important topic in education and there should be more technology transfer. The interviews indicate that code quality should play a more important role in university studies because even though all of the interviewees studied something with computer science, studies were rarely named as source of information. More often it is even said that code quality was mentioned briefly and not really discussed in the lectures. Research literature is never named which indicates that more work has to be done to transfer research results to practice. Maybe the university education is a place to start.

A big problem that many teams seem to have, is documenting standard solutions. Often, there are multiple solutions for the same problem and if an experienced developer does not review the code, these code sections are not detected and conceptual integrity is not ensured. The question is, how one can write them so even new developers can find them or at least know they exist. There does not seem to be a good solution for documenting standard solutions to ensure conceptual integrity. A more appropriate solution could be further researched.

The team's organizational environment and processes play a big part, too. A team size of six to eight practitioners is aspired, but companies often have trouble finding new developers. The bigger the team, the more communication has to take place and the bigger teams are also the ones using approaches like SCRUM. This often leads to better code quality because enforcement steps are made mandatory. Even though all interviewees' software projects are built for long-term use, it does not mean that the longevity is always kept in mind. The company size plays a role for the tools as some of them are preset. However, this does not imply more communication and knowledge transfer between the teams to make use of more experts. The interviews show that agile approaches help with code quality and should be considered. Furthermore, the company should support exchange between the different teams.

Surprisingly, the effects of tools are not only positive. Too many warnings lead to de-motivation and the effort to keep the tools running and up-to-date are seen as inconvenient. Surprisingly, the costs are not even the main factor for not using tool support. The initial effort is the key reason why new tools are not used. Rules have to be defined first to ensure that not too many warnings appear. The problems with compatibility when upgrading can be eliminated by using tools from the same company which also helps to approach the problem of obsolete technology. This should show tool providers that additional support in the beginning is needed or they should provide a detailed documentation to ensure an easy set up. Furthermore, they could differentiate themselves more clearly to their competitors to make the choice for a practitioner clearer. Furthermore, it shows that the company should give developers more time to invest into new tools.

SECTION VI.

Threats to Validity

To ensure the validity of a study, Gasson [12] proposes criteria such as confirmability, auditability, internal consistency and transferability. They help to evaluate research rigor and quality in involving qualitative research methods.

Confirmability was addressed by repeatedly discussing and restructuring the identified categories with a second researcher. In terms of auditability, a standardized research process was followed and approved methods for data analysis were used, such as the qualitative content analysis method as described by Mayring [7]. Finally, transferability was addressed by capturing and describing the background of the study participants. 12 experts were interviewed with different degrees and from different fields. The experienced varied between under a year up to 36 years of professional experience in software engineering. Another researcher was involved in the coding process to ensure internal consistency by discussing and crosschecking the codes.

This study has the following shortcomings. There are only a limited number of samples, similar to many qualitative studies. This number was sufficient, because the aim was to identify reasons why code quality is not ensured in practice, not to evaluate existing knowledge. Additionally, no new concepts emerged in the last three interviews as shown in the supplementary information1.

SECTION VII.

Related Work

There are no empirical studies concerning the lack of acceptance of Clean Code in practice. Stevensonet al. [13] state that “there does not appear to be much research exploring the techniques that software developers actually use in their day-to-day practice and which of these techniques they find most useful.” (p. 2) There is literature about evaluating static code analysis [14]–​[16], process improvements [17], [18] and the general benefits of quality control [19], [20], but there is only scarce literature about the analysis investigating which steps are actually used and where the challenges are.

The paper [13] by Stevenson and Wood is the most similar to our research. Here, the main question is, what is used by practitioners and what they find useful. Their research is more about quantity and specifically object-oriented programming. They found a strong connection of practitioners experience and design quality. It not only influences the developer's work itself, but also the effectiveness of the review process. The participants also stated that good naming, small size and simplicity contribute to good quality design. Time pressure was named as a reason for not investing enough time on design quality but there were not many questions concerning challenges in general. The participants were asked about design patterns and traditional design guidelines like coupling, cohesion and inheritance. That may be the reason why they were viewed more important compared to the interviews where the participants were not specifically asked about them.

In the literature, the term software process improvement (SPI), is used when discussing how to implement process improvements in companies. With the ISO Standard or Capability Maturity Model (CMM) the current quality of the processes can be assessed and improved. Baddoo and Hall [17] discuss de-motivators that were named by developers, product managers and senior managers. The challenges are similar to the ones named during the interviews. Time pres-sure/constraints were named most often and others like inertia, lack of standards, inexperienced staff and lack of overall support were named, too. It is questionable if code quality practices can be compared to SPI because SPI is more about improving the processes and not the code quality. However, they both want to achieve a better software systems in the end.

Beecham et al. [21] show a systematic literature review about motivation in software engineering. They focus on motivators for developers to stay at their current job, but they “[do] not shed light on what motivates an individual […] to produce better quality software, increase productivity, and use and share skills in the wider Software Engineering community.”

Steidl et al. [19] show the importance of continuous quality control. They found that not only automated measurements are necessary but also human evaluation and interaction to ensure code quality. They used ten rules that were checked frequently like code redundancy, structural criteria and test coverage over a period of up to 5 years. It is not clear whether there were additional improvements performed in the time period. Miller et al. [20] compare the usefulness of checklist and scenarios and underline the importance of inspection. There seems to be more support for the scenario approach than for the checklist. The generalization here is questionable because mostly students were questioned and as they said, there is “a strong correlation between general academic ability and the ability to successfully inspect software.” [20, p. 57].

Developers, team leaders, project managers and architects were interviewed by Yamashita and Moonen [22] to find out if developers care for code smells. Even though it was asked when and how they fixed code smells, the answer was not discussed in the paper. They “found that software professionals who are interested in code smells and anti-patterns expressed a need for better support during the software evolution cycle” (p. 9). Surprisingly 32 % of the respondents did not know about code smells. It was clearly stated whether they just did not know the term or actually were not aware of them. The participants of this study all named some when asked about frequent violations. Duplication, length of methods, large classes and long parameter lists were all also named in the interviews. The concerns varied slightly, as the developer's skills do not seem to be of high concern, while the participants expressed a need for further training not only for themselves but for their whole team. The sources of information were similar with blogs, forums, books and colleagues.

SECTION VIII.

Conclusion, Implications and Future Work

In this paper, the results from an empirical study were presented. Problems and challenges were investigated that occur while enforcing Clean Code. Twelve practitioners from different companies and backgrounds were interviewed which were enough to saturate the research. The semi-structured interviews were analyzed using open coding. Detailed feedback was gathered and as a result, important insight was gained about enforcement of Clean Code in practice and about successful introduction of new tools and processes.

There are multiple reasons for practitioners not paying enough attention to code quality. Time and management pressures are not the main causes of the lack of attention for Clean Code principles. Without defined standards and rules, the monitoring is difficult to perform. Low priority and subjective aspects of code quality make the evaluation difficult. Surprisingly, tools and guidelines were not seen as helpful in this case. Tools took too much initial effort, and guidelines were too vague and not easily controllable. The main effort to improve code quality should go to the four people-related aspects motivation, knowledge, experience and code quality awareness as they play a key role in achieving good quality. The initiative of practitioners is often the reason why code quality even becomes a topic in a company. The attitude in the team towards Clean Code is an important factor for a later introduction of new tools and conventions.

Implications for Academic Research

This paper showed the need for further research concerning enforcement of code quality in practice. There should also be more emphasis on the people-related aspects instead of just new metrics and tools. The best way to document standard solutions could be further researched, too. The effectiveness of guidelines is another possible research topic. Finally, researchers should find ways how practitioners could benefit from their results.

Implications for Practitioners

Generally, practitioners should be able to identify the problems in their teams more easily. There are some implications concerning organizational aspects. More time should be given to invest into new tools and the priority of code quality should be reconsidered. Furthermore, any company should support the exchange between different teams. The results show that experienced and knowledgeable team members are very important.

The interviews also show that guidelines have to be defined more clearly and more easily controllable. Agile development processes should be considered because they support better code quality enforcement. Steps and processes like code review, refactorings and testing should be made mandatory so the developers are not inclined to drop them. When doing pair programming, practitioners should evaluate the need and effectiveness to ensure resources are not wasted. But it should also not be omitted all together.

Implications for Tool Providers

The initial effort for new tools is the strongest obstacle. Tool providers should offer additional support in the beginning or provide a detailed documentation to ensure an easy setup. Furthermore, they should differentiate themselves more clearly from their competitors to allow a clearer choice for practitioners. With this study, we provide a first step towards improved support of ensuring code quality in practice.

The next step is to look into the improvement of code quality acceptance and enforcement. We aim to develop an approach to further improve these two aspects.
